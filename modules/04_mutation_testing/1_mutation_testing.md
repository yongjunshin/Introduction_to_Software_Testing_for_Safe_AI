## **What is Mutation Testing?**

### **From Structural Testing to Mutation Testing**

Structural coverage is based on the rationale that **a test suite that executes more of the code is more likely to reveal faults**. However, coverage is only an indirect indicator of defect detection quality. 

**Mutation testing** changes the perspective by asking a more direct question: **how good is this test suite at detecting faults artificially injected into the program?**

### **Definitions**

**Mutation testing**: a fault-based testing technique that evaluates the effectiveness of a test suite by creating many slightly modified versions of the program, called **mutants**, each containing a small syntactic change (a potential fault), and checking whether the tests can detect these injected faults (i.e., cause the mutants to fail).

**Mutant**: a new version of the program created by applying a small syntactic change that represents a potential fault.

- Mutants are generated by applying simple mutation operators (e.g., changing `>` to `>=`, `+` to `-`, constant values, logical operators).
- A mutant is **killed** if at least one test case produces a different outcome on the mutant than on the original program.
- A mutant is **survived** if all tests pass on both the mutant and the original program (i.e., the test suite fails to distinguish them).

**Mutation Score**: A metric of how effective a test suite is at killing mutants.

- Formally, for a set of mutants \(M\), the mutation score is:
  $
  \text{Mutation Score} = \frac{\text{Number of killed mutants}}{\text{Number of non-equivalent mutants}}
  $
- **Equivalent mutants** are mutants that behave identically to the original program for all possible inputs (i.e., no test can ever kill them); they should be excluded from the denominator when possible.
- A higher mutation score means the test suite is better at detecting injected faults and is therefore considered more thorough and powerful.

## **Mutation Testing for Traditional Software**

### **Overall Process**

High-level workflow of mutation testing: generate mutants, run the existing test suite against them, compare outputs with the original program, and use the results to assess and improve the test suite.

### **Mutation Operators**

Common syntactic changes (e.g., arithmetic, relational, logical, and constant modifications) that simulate typical programmer mistakes and are used to create mutants.

| Language | Mutation Testing Tools                                  |
|----------------------|-----------------------------------------------------------------|
| C / C++              | Mull, MiLu                                                 |
| Java                 | **MuJava<sup id="fnref1">[1](#fn1)</sup>**, PIT, Major                               |
| JavaScript           | StrykerJS                               |
| Python               | mutmut, Cosmic-Ray, MutPy                                |

## **Mutation Testing for DL Software**

### **Source-level Mutation Testing**

Applies mutation operators to the source code around deep learning components (e.g., data preprocessing, training logic, loss functions) to evaluate how well tests detect faults in DL pipelines.

### **Model-level Mutation Testing**

Directly mutates elements of a trained deep learning model (e.g., neuron weights, activation functions, network structure) to study whether test inputs can detect changes in the model’s behavior.
